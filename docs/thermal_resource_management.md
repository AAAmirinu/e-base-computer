# 温度、精度、動的分割数の資源管理

## 1. 基本発想

Eセルが `0 <= d < e` の連続値を保持できるとしても、実用機では温度や外乱に
よって値が揺らぐはずです。

このとき、離散モードで安全に使える分割数は固定ではありません。
冷えていて安定したEセルは細かく分割できますが、熱いEセルは粗くしか
分割できません。

つまり、E進コンピューターではメモリ容量が単に「何セルあるか」ではなく、
次のように変動します。

```text
利用可能な情報量 = Eセル数 * 現在安全に読める分割数
```

より正確には、分割数そのものではなく、その対数、つまり
「何個の三値桁に相当するか」が資源量になります。

```text
三値相当桁数 = floor(log_3(安全分割数))
```

この発想はかなりE進コンピューター独特です。
普通のコンピューターでは熱くなると遅くなりますが、E進機械では
**熱くなるとメモリの意味解像度が落ちる** ことがあります。

## 2. 安全分割数

Eセルを `q` 個の離散状態として使う場合、各状態の間隔はおおよそ次です。

```text
delta = e / q
```

一方、温度による揺らぎ、読み出し誤差、経年劣化、周辺セルからの干渉を
まとめた安全余白を `guard(T)` とします。`T` は温度です。

安全に `q` 分割するには、隣の状態と十分に離れている必要があります。

```text
delta > 2 * guard(T)
```

したがって、温度 `T` での最大安全分割数は概念的に次で表せます。

```text
q_max(T) = floor(e / (2 * guard(T)))
```

これは厳密な物理式ではなく、作中工学で使う安全設計式です。
温度が上がるほど `guard(T)` が大きくなり、`q_max(T)` は小さくなります。

## 3. 分割数の階段

実用機では、任意の分割数を連続的に使うより、決まった段階へ丸めるはずです。

三値論理と接続しやすい階段:

```text
3値   = 三値1桁
9値   = 三値2桁
27値  = 三値3桁
81値  = 三値4桁
243値 = 三値5桁
```

ただし、すべてのEセルが243値まで使えるとは限りません。
高価な低温Eメモリだけが細かい分割を許され、一般のEメモリは3値から27値
程度に制限される、という棲み分けが自然です。

## 4. コア桁と余剰桁

温度で分割数が落ちるなら、保存する情報を2層に分けるのが有効です。

- **コア桁**: 温度が多少上がっても保持される重要な三値桁。
- **余剰桁**: 冷えているときだけ使える追加情報。

例えば、27値モードのEセルを三値3桁として使う場合:

```text
最下位の1桁: コア桁
上位の2桁: 余剰桁
```

温度が上がって27値を維持できなくなったら、余剰桁を捨てて3値モードへ
退避できます。これは、画質を落としても内容は残る画像圧縮に少し似ています。

作中用語としては、次のように呼べます。

- **熱耐性桁**
- **揮発桁**
- **冷却桁**
- **影桁**

## 5. 熱による降格

Eセル列が熱くなると、OSはその領域をより粗いモードへ降格します。

```text
81値 -> 27値 -> 9値 -> 3値 -> 退避
```

このとき重要なのは、単に情報が壊れるのではなく、
**意味が粗くなる** という点です。

連続値なら、有効桁数が減る。
多値記号なら、細かい状態がまとめられる。
三値パックなら、余剰桁が失われる。

この挙動は物語上も使いやすいです。
「データが消えた」のではなく、「機械が熱で世界を粗く見始めた」と表現できます。

## 6. OSの資源管理

E進OS、またはE状態を管理するカーネル層は、次の資源を扱います。

- Eセル数。
- 温度。
- 冷却能力。
- 現在の安全分割数。
- 要求される最低分割数。
- 余剰桁を捨ててよいか。
- 観測による追加発熱。

普通のOSがCPU時間やメモリ量を割り当てるように、E進OSは
**精度予算** と **冷却予算** を割り当てます。

例:

```text
航法計算:
  最低27値、冷却優先度 高、余剰桁破棄 不可

娯楽用映像生成:
  最低3値、冷却優先度 低、余剰桁破棄 可

AIの感情モデル:
  最低9値、冷却優先度 中、連続値劣化 許容
```

この世界では、重い処理とは「計算量が多い」だけでなく、
**冷たいEセルを長時間占有する処理** でもあります。

## 7. 熱ページング

通常のOSには、メモリが足りないとストレージへ退避するページングがあります。
E進OSには、温度や精度に応じた **熱ページング** がありそうです。

熱ページングの例:

- 重要なE場を冷却済みバンクへ移動する。
- 重要でないE場を27値から9値へ降格する。
- 連続状態を低精度スナップショットへ退避する。
- 三値素子メモリへコア桁だけ保存する。
- 熱が下がったあと、余剰桁を再生成または再学習する。

これにより、E進コンピューターは単に「メモリ不足」ではなく、
「冷却された高精度領域が不足」することになります。

## 8. プログラム側の指定

E進プログラムは、値に対して最低精度を指定する必要があります。

型の例:

```text
ETrit[min=3]
EPackedTrit[min=27, degrade=allow]
EContinuous[epsilon=1e-6, thermal_class=cold]
EField[min_partition=9, surplus=discardable]
```

`degrade=allow` が付いた値は、温度上昇時に粗い表現へ落としてもよい値です。
一方、`degrade=deny` の値は、精度を維持できない場合に例外を起こします。

この仕組みにより、E進プログラミングでは「どの値が劣化してもよいか」を
型として書く文化が生まれます。

## 9. 例外と警告

熱と分割数に関係する例外は、次のようになります。

- **分割数降格**: 27値として使っていたE場が9値へ落ちた。
- **最低精度違反**: プログラムが要求した分割数を維持できない。
- **余剰桁消失**: 冷却時だけ使える桁が失われた。
- **熱観測拒否**: 読み出すと発熱して精度が落ちるため、観測が拒否された。
- **冷却待ち**: 高精度計算のため、E場が冷えるまでスケジュールが止まる。

このあたりは作中の緊張感にも使えます。
例えば戦闘中、航法炉が熱で27値から9値へ降格し、正確な軌道予測ができなくなる。
ただし最低限の3値制御は残るので、完全停止ではなく「荒い判断しかできない」
状態になります。

## 10. セキュリティへの影響

温度によって分割数が変わるなら、攻撃者は冷却や加熱を通じて情報を漏らしたり、
逆に潰したりできるかもしれません。

攻撃例:

- 局所加熱で余剰桁を消す。
- 冷却して、本来読めない細かい状態を観測する。
- 温度変化を測って、E場がどの程度の分割数で使われているか推測する。
- 観測による発熱を利用して、他プロセスの精度を落とす。

そのため、E進OSの観測権限は温度管理とも結びつきます。
「読む権利」だけでなく、「どの温度条件で、何分割まで読んでよいか」を
許可しなければなりません。

## 11. 世界設定としての言い回し

この設定が入ると、作中の技術者は次のような言い方をしそうです。

- 「このE場はまだ27まで読める」
- 「冷却桁を捨てて3値で走らせろ」
- 「航法炉の精度予算を医療AIから奪う」
- 「観測すると熱で潰れる。今は見ないほうが正確だ」
- 「その記憶は消えたんじゃない。粗くなったんだ」

E進コンピューターの資源管理は、容量、速度、電力だけでなく、
**冷たさと意味解像度** を奪い合う世界になります。
