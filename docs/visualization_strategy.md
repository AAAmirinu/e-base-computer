# EPU内部動作の可視化方針

## 1. 問題意識

命令語、コンパイラ、ファイルシステム、OSは、どれも内部の複雑さを隠すための
抽象化です。

そのため、EPUエミュレーターが完成に近づくほど、利用者から見ると
「普通の仮想マシンが普通のコードを実行しているだけ」に見える危険があります。

E進コンピューターらしさを伝えるには、最終結果ではなく、
**内部で状態がどう揺れ、正規化され、観測され、劣化し、保守されるか** を
見せる必要があります。

## 2. 見せるべきもの

EPUで可視化すべき中心は、次の6つです。

- **E桁の配置**: どの指数にどの桁が入っているか。
- **Eキャリー**: 正規化で値が上位指数へ流れる様子。
- **熱状態**: 命令実行で温度が上がり、冷却で下がる様子。
- **許可分割数**: 温度に応じて `243 -> 81 -> 27 -> 9 -> 3` と落ちる様子。
- **モード変化**: `EWORD`, `CONTINUOUS`, `PACKED_TRIT`, `OBSERVED` の切り替え。
- **観測の影響**: `EOBS` により `OBSERVATION_DIRTY` が立つ様子。

これらは、通常のCPUエミュレーターでは主役になりません。
ここを前面に出すことで、EPUが単なる算術VMではないことを伝えられます。

## 3. 可視化ビュー

### 3.1 命令タイムライン

各命令を1行にして、実行前後の差分を表示します。

```text
tick  op      target  flags
0     ECONST  ER0     NORMALIZED
1     EMUL    ER2     NORMALIZED, thermal +0.08
2     EQUANT  ER5     QUANTIZED, partition 27
3     EOBS    OUT0    OBSERVATION_DIRTY
```

これは、普通の命令トレースに見えますが、フラグにEPU固有の意味を持たせます。

### 3.2 E桁ラダー

EWordを指数方向のラダーとして表示します。

```text
e^3 | 2.6449
e^2 |
e^1 |
e^0 |
```

正規化前後を並べると、Eキャリーが「上へ流れる」様子を見せられます。

### 3.3 セル環ビュー

1つのEセルを `[0, e)` の円環または横ゲージとして表示します。

```text
0 |---- state 0 ----| guard |---- state 1 ----| guard | ... | e
```

離散モードでは安全域とガードバンドを色分けします。
温度が上がるとガードバンドが広がり、使える分割数が減る様子を表示します。

### 3.4 熱マップ

EレジスタやEメモリバンクを、温度で色分けします。

```text
ER0 cool
ER1 warm
ER2 hot
COLD bank stable
WORK bank unstable
```

このビューは「冷たいEセルを奪い合う」という資源管理を直感的に見せます。

### 3.5 Eメモリ場ビュー

Eメモリ上の確保領域を、場として表示します。

```text
COLD bank
[F0:EWORD:27] [F1:PACKED_TRIT:9] [free...]
```

普通のメモリビューではなく、モード、分割数、温度、リフレッシュ期限を
一緒に表示します。

### 3.6 観測ビュー

`EOBS` は普通のprintではなく、観測イベントとして表示します。

```text
ER3 was observed as OUT0=144.40872214
flag: OBSERVATION_DIRTY
temperature: +0.03
```

「見たこと」が状態に影響する、というE進らしさを強調します。

## 4. エミュレーター側に必要なフック

可視化UIを作る前に、EPUエミュレーターは次を出せる必要があります。

- 実行前スナップショット。
- 実行後スナップショット。
- 命令名、オペランド、オプション。
- 状態フラグ。
- レジスタごとの値、モード、温度、分割数。
- E場ごとのバンク、範囲、モード、温度、分割数。
- 観測出力。
- 例外情報。

このため、EPU本体には **構造化イベントログ** を持たせます。

## 5. イベントログ形式

イベントログは、JSONに近い辞書列として扱います。

```text
{
  tick: 12,
  op: "EQUANT",
  args: ["ER5", "ER4", "27"],
  before: { ... },
  after: { ... },
  flags: ["QUANTIZED"],
  exception: null
}
```

この形式にしておけば、後で次のどれにも使えます。

- コンソール表示。
- HTML可視化。
- ターミナル上の表。
- Webアプリ。
- 教材用のステップ実行。

## 6. 見た目の方向性

E進コンピューターは、普通のCPUよりも「場を調律している」印象を出したいです。

そのため、UIは以下を重視します。

- クロックよりもtickと温度の推移を見せる。
- メモリよりもE場を見せる。
- ビット列よりも `[0, e)` の連続ゲージを見せる。
- 例外よりも「降格」「観測汚れ」「再正規化」を見せる。
- 最終値よりも、途中の意味変化を見せる。

## 7. 実装順序

まずは、GUIではなくエミュレーターの出力を強くします。

```text
1. EPUに構造化イベントログを追加する。
2. 実行前後のスナップショットを取る。
3. デモでイベントログの一部を表示する。
4. テストでイベントログが生成されることを確認する。
5. その後、HTMLまたはCLIの可視化ビューを作る。
```

この順番なら、UIを作る前に「何を見せるべきか」がコード側で固まります。
