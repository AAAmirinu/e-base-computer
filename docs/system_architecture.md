# E進コンピューターのシステムアーキテクチャ

## 1. 結論

E進コンピューターに必要になる独特の仕組みは、普通のCPU、メモリ、OSを
完全に置き換えるものではありません。

より自然なのは、次のような混成構造です。

```text
二値・三値の制御計算機
  -> 命令発行、保護、通信、通常のOS処理

E進演算炉
  -> 連続値、対数スケール、指数基底の計算

境界装置
  -> 量子化、正規化、誤差監視、Eセルの意味切り替え
```

つまり、E進コンピューターは「E進CPUだけで動く機械」というより、
**通常のデジタル計算機が、E進演算炉を管理する機械** と考えると
説得力があります。

## 2. CPU

### 2.1 通常CPUとE進実行ユニット

この世界の高級機には、通常CPUの横に **EPU**、つまり
**E Processing Unit** が載っていると考えられます。

通常CPU:

- OSを動かす。
- プロセスを管理する。
- 入出力、通信、暗号、ファイル操作を行う。
- EPUへジョブを投げる。

EPU:

- E進ワードの加算、乗算、正規化を行う。
- Eセル列を連続値として演算する。
- 対数シフト、畳み込み、指数基底変換を行う。
- 必要に応じてEセルを三値・多値モードとして扱う。

現実のGPUやNPUに近い関係です。ただしEPUは、単なる高速行列計算器ではなく、
連続値を物理的に保持したまま計算するところが違います。

### 2.2 EPUに必要な命令

EPUの命令セットには、普通のCPUにはない命令が必要になります。

- `ENORM`: Eキャリーを伝播させ、E進ワードを正規化する。
- `ESHIFT n`: 指数を `n` だけずらし、値を `e^n` 倍する。
- `ECONV`: E進ワード同士を畳み込み乗算する。
- `EQUANT q`: Eセル値を `q` 個の離散状態に量子化する。
- `EDEQ`: 離散状態をEセル上の代表値へ戻す。
- `ECLAMP`: 境界に近い値を安全域へ押し戻す。
- `EREFRESH`: Eセル列を校正し、正規化し、健全性メタデータを更新する。
- `ETHERM`: Eセル列の温度、揺らぎ、許容分割数を読む。
- `EQOS`: 必要な精度、最低分割数、冷却優先度を要求する。
- `EMODE`: Eセル列の意味を連続、三値、多値、係数列などへ切り替える。
- `EOBS`: E進値を通常の数値表現へ観測・変換する。

この中で特にE進らしいのは `ENORM`, `ESHIFT`, `EMODE` です。

エミュレーター実装に向けた命令形式、レジスタ、Eメモリ、例外の詳細は
`docs/epu_spec.md` で定義します。

### 2.3 レジスタ

EPUのレジスタは、単純な固定長ビット列ではなく、次の情報を持ちます。

- Eセル列。
- 指数範囲。
- 符号。
- 現在の解釈モード。
- 精度またはガードバンド設定。
- 誤差履歴。

作中用語としては、通常のレジスタではなく **Eレジスタ**、
大きなものは **E束** や **場レジスタ** と呼んでもよいです。

## 3. 記憶装置

### 3.1 Eセルはメモリとしては高価

Eセルは連続値を保持できるため、記憶装置としても使えます。
しかし、大量の一般データを置くには高価です。

そのため、記憶階層は次のようになるのが自然です。

```text
通常ストレージ:
  文書、プログラム、ログ、暗号鍵、複製可能なデータ

三値または二値メモリ:
  OS、制御状態、命令列、通常の作業メモリ

Eメモリ:
  連続値状態、物理モデル、係数場、AIの内部状態、観測中の信号
```

### 3.2 Eメモリは「保存」より「状態保持」

Eメモリは、普通のストレージというより **状態を生かしたまま置いておく
作業領域** です。

例えば、気象制御AIが大気の状態をEセル列として保持している場合、
それはファイルというより「低速で変化し続ける計算対象」です。

このためEメモリには、普通のメモリにはない概念が必要になります。

- **冷却保存**: 連続状態の揺らぎを抑える。
- **再正規化周期**: 一定時間ごとにEキャリーと誤差を整理する。
- **観測コスト**: 読み出しそのものが状態を荒らす可能性がある。
- **意味タグ**: そのEセル列が連続値なのか、多値記号なのか、係数列なのかを示す。

### 3.3 Eセルのアドレス指定

通常メモリはバイト単位で番地を持ちます。
Eメモリでは、番地に加えて指数範囲とスケールが重要になります。

```text
E領域ID
開始セル
セル数
指数オフセット
解釈モード
ガードバンド幅
```

同じEセル列でも、指数オフセットを変えると意味する実数スケールが変わります。
そのため、Eメモリのポインタは普通のアドレスより重い構造になります。

作中ではこれを **Eポインタ** または **スケール付きポインタ** と呼べます。

## 4. OS

### 4.1 E進OSは必要か

完全に独立した「E進OS」が必要かというと、必ずしもそうではありません。

ただし、通常OSに次の機能は必要になります。

- EPUのジョブスケジューリング。
- Eメモリ領域の確保と解放。
- Eセルのモード管理。
- 温度に応じた許可分割数の管理。
- Eリフレッシュ周期の管理。
- 正規化周期の管理。
- 誤差、揺らぎ、ガードバンド違反の監視。
- 通常プロセスとE進ジョブの同期。
- 観測と量子化の権限管理。

つまり、E進機械で独特なのはOSそのものより、
**E状態を資源として管理するカーネル層** です。

### 4.2 プロセスではなく「場」

普通のOSは、プロセスを基本単位として扱います。
E進計算では、長時間存在する連続状態が重要なので、
OSは **E場** という資源を管理することになります。

E場:

- Eセル列からなる連続状態。
- どのプログラムが読めるか、書けるかを持つ。
- 時間とともに変化する。
- 定期的に正規化される。
- 観測すると離散値へ写されることがある。

これは、ファイル、メモリ、プロセス、デバイスの中間のような存在です。

### 4.3 観測権限

Eセルは連続値なので、読み出し方によって意味が変わります。
同じE場を、あるプログラムは連続値として読み、別のプログラムは27値の
離散記号列として読むかもしれません。

そのためOSは、単なる読み書き権限だけでなく、**観測権限** を管理します。

- 連続観測を許可するか。
- 離散観測だけ許可するか。
- 何分割まで許可するか。
- 観測後に元のE場を乱してよいか。
- 観測結果を暗号的に信頼できる形で記録するか。

これはE進コンピューター独特の、とてもおいしい設定です。
「アクセス権はあるが、連続観測権がない」というセキュリティ概念を作れます。

### 4.4 熱と精度のスケジューリング

Eセルの温度によって揺らぎが変わるなら、OSは計算時間やメモリ容量だけでなく、
**現在そのE場を何分割まで安全に読めるか** も管理します。

例えば、冷えているE場では27値モードが許され、少し温まると9値モードへ、
さらに温まると3値モードへ落ちる、という挙動です。

この管理は `docs/thermal_resource_management.md` で詳しく扱います。

### 4.5 長期保存とEリフレッシュ

Eセルの連続値を長期保存するには、DRAMのリフレッシュに似た定期保守が
必要になります。ただしE進機械の場合、単に値を書き直すだけではなく、
温度、揺らぎ、正規化、量子化安全域、参照セルとのずれをまとめて点検します。

この操作を **Eリフレッシュ** と呼びます。詳細は
`docs/long_term_storage.md` で扱います。

## 5. ファイルシステム

E進世界のファイルシステムには、普通のファイルとは別に
**Eスナップショット** が存在すると考えられます。

通常ファイル:

- ビット列または三値列。
- 完全に複製できる。
- ハッシュで同一性を確認できる。

Eスナップショット:

- E場の状態を近似的に凍結したもの。
- 精度、量子化幅、観測条件がメタデータとして付く。
- 完全コピーではなく、許容誤差つきの複製になる。

そのため、Eファイルには次のメタデータが必要です。

- 基準時刻。
- 温度または安定化条件。
- 量子化方式。
- 指数オフセット。
- 許容誤差。
- 再生に必要な正規化規則。

この世界では、「ファイルを開く」と「状態を蘇生する」が少し近い概念に
なっているかもしれません。

## 6. コンパイラとプログラミング言語

E進コンピューターでは、プログラムは単に命令列を書くのではなく、
値がどの解釈モードで存在しているかを指定する必要があります。

型の例:

```text
EContinuous[range=-8..8]
ETrit[digits=3]
EPackedTrit[count=27]
ECoefficientBasis[lambda_set=...]
EObservedFloat[precision=...]
```

コンパイラは次を管理します。

- 連続モードと離散モードの切り替え。
- 量子化のタイミング。
- 正規化の挿入。
- ガードバンド不足の検出。
- 通常CPUとEPUの処理分担。

人間のプログラマが全ての `ENORM` や `EQUANT` を手で書くのは辛いので、
E進コンピューターの実用化には、かなり賢いコンパイラが必要です。

## 7. 割り込みと例外

E進機械には、普通のCPU例外に加えて、独特の例外がありそうです。

- **正規化過負荷**: キャリー伝播が想定より広がった。
- **境界接近**: Eセル値が離散状態の境界へ近づきすぎた。
- **観測破壊**: 観測によりE場の精度が落ちた。
- **意味衝突**: 同じE領域を連続モードと離散モードで同時に使おうとした。
- **スケール逸脱**: 指数オフセットが安全範囲を超えた。
- **再現不能状態**: 保存されたEスナップショットを許容誤差内で復元できない。

これらは物語上も使いやすいです。
例えば、宇宙船の航法炉で「境界接近警告」が出ると、単なるエラーではなく、
連続計算が離散判定へ落ちる直前の危険な状態を示せます。

## 8. 入出力

外界のセンサーは連続量を測るので、Eセルとは相性がよいです。

得意なI/O:

- 重力波、電磁場、気象、流体、生体信号。
- 非常に広いダイナミックレンジを持つ観測。
- 成長、減衰、振動、拡散を含む信号。

苦手なI/O:

- 厳密な帳簿データ。
- 暗号鍵。
- 法的記録。
- 完全一致が必要な通信。

つまりE進コンピューターは、世界を「ファイル」よりも「場」として受け取る
機械です。これは普通のコンピューターとの大きな違いになります。

## 9. 独特の必要物まとめ

E進コンピューター独自に必要になるものは、次のあたりです。

- EPU。
- Eレジスタ。
- Eメモリ。
- Eポインタ。
- E場。
- 正規化器。
- 量子化器。
- ガードバンド管理。
- 観測権限。
- Eスナップショット。
- E状態を扱えるコンパイラ。
- E状態の例外処理。
- 熱状態に応じた動的分割数管理。
- Eリフレッシュと長期保存管理。

逆に、通常のCPU、通常メモリ、通常OS、通常ファイルシステムは消えません。
E進システムは、それらを土台にして、連続状態を扱うための新しい層を
追加します。

## 10. 世界設定としての見え方

一般人にとって、E進コンピューターは普通のパソコンではありません。
むしろ次のような印象を持たれそうです。

- 起動するのではなく「安定化する」。
- ファイルを読むのではなく「状態を観測する」。
- 計算を止めるのではなく「場を凍結する」。
- バグるのではなく「意味がずれる」。
- 再起動するのではなく「再正規化する」。

この言葉遣いを作中で使うと、E進コンピューターが普通のデジタル機械とは
違う文化を持っていることを表現できます。
